# Research Landscape

> This document tracks the evolving state of the field as researcher reads papers.

## Overview

Decision agent 技术路径综述。关注决策架构的设计模式、技能管理机制，以及多智能体协同完成长期任务的方法。

## §1 Decision Agent Architecture

### §1.1 Co-Evolving Multi-Agent Systems

- **COS-PLAY** [1] (Wu et al., 2026) — 将决策智能体（A_D）和技能库智能体（A_S）解耦并用 GRPO 联合训练，通过 7–25 轮协同进化达到收敛。8B 模型（Qwen3-8B）在四款单人游戏上平均奖励超 GPT-5.4 25.1%；Diplomacy 多人游戏优于 Gemini-3.1-Pro 8.8% [1: §5.1]。关键约束：技能库分布与决策策略分布必须对齐——错位版本反而降低性能 [1: §5.2]。技能可复用性：每技能平均 12–49 次实例化，2–6 次合约精化 [1: §5.3]。
  - *builds-on* Voyager (Wang et al., 2023) — COS-PLAY 将 Voyager 的外部技能库扩展为一个独立训练的技能库智能体 [med]
  - *competes-with* SkillRL (Xia et al., 2026), SCALAR (Zabounidis et al., 2026) — 两者同样将技能学习与 RL 结合，但技能库均为静态基础设施而非独立训练的协作智能体 [med]
  - *relation to thesis* — 直接实例化"multi-agent workers team"架构；技能库智能体跨 episode 无状态运行（无持续上下文），为决策智能体独立部署提供初步支持证据 [1: §Relations]

> *Emerging sub-cluster: Graph-based Test-time Routing* — 在推理时通过有向图结构进行智能体选择与消息传递，无需 RL 训练。GoA 不适配 §1.1（Co-Evolving），放置于父桶 §1；建议增设 §1.2，详见 `contradictions.md §Proposed taxonomy extension`。

- **GoA** [2] (Yun et al., 2026, ICLR 2026) — 将多智能体协作建模为有向图，分四阶段执行：①元 LLM 从模型卡采样 top-k 相关智能体（节点采样）；②各智能体互相评分生成有向相关权重边（边采样）；③双向消息传递（高相关节点→低相关节点，再反向）；④图池化输出（GoAMax 取最高连接节点响应，GoAMean 由元 LLM 加权合成）。GoAMax（3 agents）在 MMLU 79.18、MMLU-Pro 54.78、MedMCQA 60.04 上超越 MoA（6 agents）；GoAMean 在 GPQA 40.54、MATH 73.12 上超越 MoA；推理成本降低约 58%（11 vs. 19 LLM calls，19k vs. 56k tokens）[2: §4.2, Table 2]。关键机制：相关性有向消息传递——反转流向造成最大性能下降（–2.60 MMLU-Pro，–5.05 GPQA）[2: §4.4, Table 5]。
  - *competes-with* [1] (COS-PLAY) [low]: GoA 为零样本推理时框架，COS-PLAY 为带 RL 训练的长时序任务架构；均采用多智能体协同，但操作层级完全不同（QA 推理路由 vs. 游戏任务联合进化）
  - *competes-with* MoA (Wang et al., 2024) [high]: GoA 证明 MoA 是其退化特例（k=N、全连接均匀权重、mean pooling），并在所有主要基准上系统性超越 MoA [2: §3.3, §4.2]
  - *relation to thesis* — GoA 处理推理层智能体路由（单轮多域 QA），非决策智能体的多步任务执行；图式节点采样（元 LLM 按查询动态路由至专家子智能体）可映射为 openclaw→decision agent 调度链的轻量实现，与论题互补但不直接支持或反驳核心部署独立性主张 [2: §Relations]

> *Emerging sub-cluster: Single-Agent Future-Aware Planning* — 从理论上证明步进式推理（CoT/ReAct/Reflexion）等价于贪心策略，无法在长时序任务中替代前瞻规划。FLARE 不适配 §1.1（Co-Evolving）或图路由子簇，放置于父桶 §1；建议增设 §1.3，详见 `contradictions.md §Proposed taxonomy extension`。

- **FLARE** [3] (Wang et al., 2026) — 形式化证明步进式 LLM 推理（CoT、ReAct、Reflexion）等价于最大化局部代理分数的贪心策略，从结构上不足以支撑长时序规划 [3: §2, §3.2]。关键实证：第一步决策中，贪心策略在 55.6%、beam search 在 71.9% 的情况下选中局部最优但全局次优的"陷阱"动作，前瞻仅 23.6%；首步错误后单步恢复概率 5.4%（贪心）vs. 29.7%（FLARE）[3: §3.1, Table 3]。FLARE 通过 MCTS + 后向价值传播 + 轨迹记忆实现前瞻规划：CWQ Hits@1 71.8（FLARE）vs. 59.8（单步）[3: §5.2, Table 1]；LLaMA-8B + FLARE 频繁超越 GPT-4o + 标准推理，表明规划能力无法通过参数规模恢复 [3: §5.2]。关键局限：假设确定性转移、显式状态与规划时可用评估信号；单智能体评测，无多智能体协同证据 [3: Limitations]。
  - *orthogonal to* [1] (COS-PLAY) [med]: COS-PLAY 改进智能体"知道做什么"（技能库），FLARE 改进"如何在已知动作中选择"（规划机制）；两者解决正交缺口，组合系统需要二者 [1: §6, 3: §7]
  - *orthogonal to* [2] (GoA) [low]: GoA 处理智能体间通信路由（单轮 QA），FLARE 处理智能体内多步动作选择；系统层级完全不同，无竞争关系
  - *relation to thesis* — 为论题提供关键约束：若 decision agent workers 采用步进式推理（当前主流），在需要多步序列决策的任务上系统性失败；支持论题扩展——workers 内部规划机制必须具备前瞻性，而非贪心局部。不挑战多智能体框架，但对 openclaw×decision agent 协作栈提出单智能体内规划先决条件 [3: §Relations]

> *Emerging sub-cluster: Single-Agent Sufficiency Baseline* — 形式化证明并实证验证同质多智能体工作流可折叠为单智能体顺序执行而无精度损失，成本更低。[4] 不适配 §1.1（Co-Evolving）、图路由子簇或前瞻规划子簇，放置于父桶 §1；建议增设 §1.4，详见 `contradictions.md §Proposed taxonomy extension`。

- **OneFlow** [4] (Xu et al., 2026) — Proposition 1 形式化证明：在确定性工具副作用、基于可见历史的路由策略、共享随机性三个前提下，单 LLM 模拟器与多智能体系统的轨迹分布完全相同 [4: §3.1]。OneFlow（单智能体执行 + MCTS 双元 LLM 工作流搜索）在 6 个公开基准上匹配或超越 AFlow（多智能体），成本大幅降低：HumanEval $0.020 vs. $0.198（约 10×），HotpotQA F1 73.5 vs. 72.1，成本 $0.278 vs. $1.438（约 5×）[4: §4.2.1, Table 1–2]。单智能体 KV cache 复用将前缀成本从 Σ L_t 降至 Σ ΔL_t（渐进节省）[4: §3.1]。关键约束：等价结论仅适用于同质工作流（所有 agents 共享同一基础 LLM）；异质工作流（混合模型）性能上界由最强同质工作流决定，未超越 [4: §2, §4.2.3]；所有实验 temperature=0，形式化等价不覆盖随机采样 [4: §4.1]；最复杂基准为 TravelPlanner（单会话规划），多会话企业工作流未覆盖 [4: §4.1]。
  - *contradicts* thesis §1 (multi-agent necessity) [med]: 论题以多智能体 workers team 为架构核心；[4] 证明同质配置下多智能体结构无精度收益且增加成本——部分证伪；Decision Agent 的 BKN+ContextLoader+Dolphin+ISF 专化组件若构成异质配置则结论不适用 [4: §3.1; thesis §1]
  - *competes-with* [2] (GoA) [med]: GoA 将图结构多智能体路由作为性能路径；[4] 证明同质设置下复杂路由结构不必要——GoA 使用异质 7–8B 专家，[4] 结论不直接反驳，但将"单智能体基线"作为效率门槛施加于所有多智能体路由设计 [4: §3.1; 2: §3.3]
  - *extends* [3] (FLARE) [low]: FLARE 证明步进式推理规划不足；[4] 进一步表明多智能体任务分解工作流可折叠为单智能体顺序推理而不损精度——合并暗示性能天花板由规划机制决定而非通信拓扑，但此为合成推断，两篇论文均未明确 [3: §7; 4: §6]
  - *orthogonal to* [1] (COS-PLAY) [med]: COS-PLAY 使用独立训练 LoRA 适配器的异质双智能体，属 [4] 自身标准下必须保留多智能体结构的场景；[4] 明确指出单 LLM 模拟"无法捕获异质工作流" [4: §6; 1: §3]

> *Emerging sub-cluster: Catalog-Layer Tool/Agent Retrieval* — 在工具/智能体目录上构建类型化知识图，用类型加权 RRF 进行召回；与 §1.2（GoA 推理时图路由协同输出）层级不同——此处图是静态目录结构，用于 pre-call 选择哪些 worker 被激活。[5] 不适配既有子簇，放置于父桶 §1；建议增设 §1.5，详见 `contradictions.md §Proposed taxonomy extension`。

- **Agent-as-a-Graph** [5] (Nizar et al., 2025) — 将 MCP 服务器（agents A）和工具（T）建模为二部图 G=(A,T,E)，边记录 owner(t)=a；检索时对 A 与 T 在统一向量空间独立 top-N，再通过类型加权 RRF（αT/(k+r) 用于工具节点，αA/(k+r) 用于 agent 节点，k=60）重排，最后沿 owner 边遍历至 K 个唯一 agent [5: §3.1, §3.2, Algorithm 1]。LiveMCPBench（70 服务器/527 工具/95 问题）：Recall@5 = 0.85、nDCG@5 = 0.47，相对 SOTA ScaleMCP（0.74/0.40）+14.9%/+14.6%；最优类型权重 αA:αT = 1.5:1（轻度偏向 agent 覆盖），1:1 退化至 0.83/0.46，3:1 退化至 0.76 [5: §4.2, §4.4, Table 1, Table 3]。跨 8 种 embedding 模型（Vertex/Gemini/Titan v1-v2/OpenAI ada-3小-3大/MiniLM）平均 Recall@5 提升 19.4%（std≈0.02），Titan v2 上+28.8% [5: §4.3, Table 2]。检索来源分布：39.13% 直接来自 agent 节点，34.44% 来自工具节点经 owner 边遍历——两类型节点都对最终路由作出实质贡献 [5: §4.2.1]。
  - *builds-on* ScaleMCP / Tool-to-Agent Retrieval (Lumer et al. 2025c-d) [high]: 同组前作；[5] 在 ScaleMCP "拼接描述单向量"基线上引入类型化节点+类型加权 RRF，将多跳查询失败模式转为"显式类型节点+图遍历"修复 [5: §1, §2.1]
  - *builds-on* RRF (Cormack et al. 2009) [high]: 标准 wRRF 是出发点，[5] 改为分段式 type-conditioned 形式（Eq. 3），保留共享分母 k+r(e) 以维持类型内 RRF 排序性质 [5: §3.3]
  - *competes-with* [2] (GoA) [med]: 均为多智能体架构中的"图"，但层级不同——GoA 在推理时构图协调多 agent 输出（消息传递+图池化），[5] 在调用前对静态目录构图选 agent；二者可组合（[5] 选 K 个 agent → GoA 协同输出），双方均未引用对方（concurrent）
  - *competes-with* [4] (OneFlow) [med]: [4] 质疑同质多智能体的必要性；[5] 在多智能体框架内默认其价值并优化路由层——若 [4] 同质等价结论成立，[5] 的检索方案失去前提；若异质性成立，[5] 的类型化召回直接服务于异质 worker 选择
  - *competes-with* [1] (COS-PLAY) [low]: 二者占据同一"系统如何知道调用什么能力"的位置——COS-PLAY 通过协同进化"学到"答案，[5] 在静态目录上"检索"答案；研究问题正交（技能习得 vs. 工具选择），但架构角色重合
  - *orthogonal to* [3] (FLARE) [high]: 规划与检索是不同问题；[5] 假设查询/子步已给定，仅负责选 agent，[3] 关注子步如何正确生成；可在同一栈共存（planner→[5]→worker）
  - *relation to thesis* — 直接命中 ContextLoader/工具管理研究缺口。论题明确将"工具检索精确召回（向量 vs 语义结构化对比）"列为高优先级；[5] 的"类型化节点显式分离 vs. 拼接单向量"恰好是该对比，14.9% Recall@5 提升构成 BKN"逻辑/行动分离"语义解耦的实证支撑。最优 αA:αT=1.5:1 提示 ContextLoader 可暴露类型权重以在意图模糊时偏向父技能/父域覆盖。规模缺口仍开放：527 工具远低于企业目标的数千级别；架构无关性（8 种 embedding）支持可移植，但未在 ContextLoader 90%+ 上下文节省真正起作用的规模上验证 [5: §Relations]

- **MCP-Zero** [7] (Fei et al., 2025) — 让 LLM 在执行中主动生成 `<tool_assistant>{server, tool}` 结构化请求触发外部检索，替代"全部 schema 注入 prompt"的被动模式。Hierarchical Vector Routing：先按 `server` 字段对 server 描述/Qwen2.5-72B 生成 summary 取较高余弦得分过滤候选 server，再在候选内对 `tool` 字段查工具描述，合成评分 `score = (s_server × s_tool) × max(s_server, s_tool)` [7: §3.1, Eq. 1]。APIBank 多轮 Full 集（Claude-3.5-Sonnet）：标准 schema 注入精度从 Domain 子集 100.00% 跌至 Full 60.22%，MCP-Zero 维持 90.32%；token 6402.2→159.0（−97.52%）[7: §5.3, Table 1]。Needle-in-haystack（1–2,797 工具，最长 248.1k token）：Claude-3.5/Gemini-2.5 上显著超越标准注入；GPT-4.1 因 baseline 已饱和无精度增益（仅省 token）[7: §5.2, Figure 5]。Q.Retrieval 基线（用户查询直接检索）仅 65–72%，作者由此推断"由模型主笔语义对齐请求"是关键收益来源而非检索本身 [7: §5.3, Table 1]。发布 MCP-tools 数据集（308 servers / 2,797 tools，commit ad2d4e6）含原始描述 + LLM 生成 server summary + 预算 OpenAI text-embedding-3-large 嵌入 [7: §4]。关键缺陷：(a) APIBank Full 仅 48 unique tools，"thousands of tools"动机与精度评测断层；(b) "LLM 写、LLM 检"循环——server summary、请求文本、目标工具描述均处 LLM 语义空间，可能放大同空间伪信号 [7: §4.1]；(c) 未与 RAG-MCP/ScaleMCP/Tool2Vec/AnyTool 等已发表工具检索方法对比，仅与 Q.Retrieval 与 Standard Tool Call 两条简单基线对照 [7: §5.3]；(d) §3.2 信息论框架（互信息最大化、O(n)→O(m+k)）从未被实验 instrumented，是装饰性事后合理化 [7: §3.2]。
  - *competes-with* [5] (Agent-as-a-Graph) [high]：同一研究问题（MCP catalog tool retrieval）下两条路径——MCP-Zero 押注"LLM 自描述请求"贴近文档；[5] 押注"显式类型节点 + 类型加权 RRF"消除多跳查询失败模式。[5] 在 LiveMCPBench (527 tools) 上将 MCP-Zero 列为基线并全面胜出（Recall@5 0.85 vs 0.70；nDCG@5 0.47 vs 0.41）[5: §4.2, Table 1]。两者可在工程上组合（MCP-Zero 请求生成 → [5] 类型化检索后端）
  - *builds-on* ReAct (Yao et al., 2023) [high]：`<tool_assistant>` 块本质是 ReAct "Thought→Action" 模式的细化变体——把"Action"再分一层"Tool Request → Tool Schema → Function Call" [7: §2.1]
  - *builds-on* Toolformer / Gorilla / HuggingGPT [med]：将 context-based tool injection 谱系归到这条线上，并把 MCP-Zero 定位为对该谱系"上下文开销不可承受"问题的回应 [7: §2.1]
  - *orthogonal to* [3] (FLARE) [med]：FLARE 关注步进推理在多步规划上的结构性失败，MCP-Zero 关注每步规划已确定后的工具召回；可在同栈共存（planner→请求生成→检索）。MCP-Zero 仅单跳检出测试，未触及多步规划；Figure 3 的 3 步链条由模型 in-context 自然分解，规划深度极浅
  - *orthogonal to* [6] (MAST) [low]：MCP-Zero 解决 FC1 系统设计的子问题（工具 schema 注入上下文压力），未触及 FC2 智能体间错位；候选少则误选少，可间接降低 FM-2.6 推理-动作错位概率，但论文未做该方向的失败模式量化
  - *orthogonal to* [4] (OneFlow) [low]：MCP-Zero 完全在单 agent 范畴内（一个 LLM + 检索器），与 OneFlow 单 LLM 模拟立场一致——是其工具供给侧实现
  - *orthogonal to* [2] (GoA) [low]：GoA 处理推理时多 agent 输出协调，MCP-Zero 处理单 agent 调用前工具供给；层级正交
  - *orthogonal to* [1] (COS-PLAY) [low]：COS-PLAY 协同进化技能库（学得能力），MCP-Zero 在静态 catalog 上检索已有工具（选择能力）；问题域不同
  - *relation to thesis* — 直接命中"工具幻觉的语义解耦防御"研究缺口（thesis §设计上下文 #3）。三点对 Decision Agent 的可操作启示：(a) "模型主笔请求 vs 用户查询直接检索"在精度上重要——APIBank 多轮 Full 上前者 90.32% vs 后者 65.05%，与 BKN "逻辑/行动分离"思想同向，可作为 ContextLoader 前置请求改写器的可行范式；(b) 规模缺口仍在原位——APIBank Full 仅 48 工具，远低于企业目标 5k–10k；2,797 工具的 needle-in-haystack 仅测"能否一次检出 1 个目标工具"，多步、跨域链式检索精度未验证；(c) 强模型上无精度增益（GPT-4.1 在 needle-in-haystack 上对 MCP-Zero 无精度收益，仅省 token），意味着"主动检索"对精度的杠杆主要在弱模型；Decision Agent 若以最强基础模型落地，活跃检索的实际增益更可能落在 token 成本侧而非精度侧——与论题"上下文窗口工程约束"的优先级一致，但需修正"主动检索能提升精度"的过度承诺 [7: §Relations]

> *Emerging sub-cluster: Multi-Agent Failure Diagnostics* — 通过对真实 MAS 执行轨迹做扎根理论编码，建立可量化的失败模式分类法，配套 LLM-as-Judge 实现规模化标注。[6] 不适配 §1.1（Co-Evolving）、图路由子簇、前瞻规划子簇、单智能体充分性子簇或目录层检索子簇——它不提出新架构，而是诊断既有架构为何失败；放置于父桶 §1，建议增设 §1.6，详见 `contradictions.md §Proposed taxonomy extension`。

- **MAST** [6] (Cemri et al., 2025) — 对 7 个开源 MAS 框架在编码/数学/常识基准上的 1642 条执行轨迹做 Grounded Theory 编码，得出 14 个细粒度失败模式分布于 3 大类：FC1 系统设计问题 44.2%、FC2 智能体间错位 32.3%、FC3 任务校验 23.5% [6: §4, Figure 1]。三大头部失败模式为 FM-1.3 步骤重复 15.7%、FM-2.6 推理-动作错位 13.2%、FM-1.5 终止条件无感知 12.4% [6: Figure 1]。关键基准失败率：AppWorld Test-C 86.7%、HyperAgent 74.7%、ChatDev 66.7%、Magentic-One 62%、MetaGPT 60%、AG2 41% [6: Table 1]。三专家在 MAST 上 Cohen's κ=0.88（域内）、0.79（Magentic-One/OpenManus 域外验证）；OpenAI o1 LLM-as-Judge 标注器 κ=0.77 [6: §3.2–3.4, Table 2]。同模型架构差异：MetaGPT vs. ChatDev 使用相同 GPT-4o，FC1/FC2 失败少 60–68% 但 FC3 多 1.56×（ChatDev 显式审阅阶段）[6: §5.1]。同架构模型差异：MetaGPT + GPT-4o 比 + Claude-3.7-Sonnet 少 39% FC1 失败 [6: §5.1, Figure 8]。MAST 指导的针对性干预：ChatDev 角色规约修复 +9.4%、增加高层目标校验步骤 +15.6% [6: §1, Appendix H]。
  - *builds-on* [4] (OneFlow) [high]: 两者并攻"MAS 表现不及预期"——[4] 在同质配置下证明多智能体结构无精度增益，[6] 在所有 7 个 MAS 中诊断出 14 类失败模式；[4] 的单 LLM 模拟器构造性消除 FC2（无智能体间通信），减少 FC1（单一终止点），与 [6] 的失败成因诊断高度互补 [4: §3.1; 6: §1, FC2]
  - *competes-with* [2] (GoA) [med]: GoA 把图式多智能体路由作为性能路径；[6] 在 7 个 MAS 中度量出 FM-2.3 任务偏离 7.4%、FM-2.6 推理-动作错位 13.2%——这两类失败随路由复杂度而扩展。[6] 未直接评测 GoA，但提供分析框架：每条新增路由边都是 FC2 失败的潜在入口 [2: §3.3; 6: §4 FC2]
  - *extends* [3] (FLARE) [med]: [3] 形式化证明步进式推理在长时序任务结构性不足；[6] 的 FM-1.5 终止条件无感知 12.4% 与 FM-3.1 过早终止 6.2% 是规划深度不足在 MAS 执行层的具体表征。[6] 未与规划理论对接，但所有 7 个 MAS 中终止类失败的高发率支持 [3] 的核心论断：瓶颈在规划机制而非通信拓扑 [3: §3.2; 6: Figure 1]
  - *orthogonal to* [1] (COS-PLAY) [med]: COS-PLAY 关注训练时的技能-决策协同进化；[6] 关注运行时失败诊断；操作层级不同，但同属"MAS 不仅是 agents 之和"主题 [1: §3; 6: §1]
  - *orthogonal to* [5] (Agent-as-a-Graph) [low]: [5] 改进调用前的工具/agent 选择机制；[6] 诊断已选定 agents 协同执行时的失败模式；可组合（[5] 提供更准确的 worker 选择 → 减少 [6] 中 FC1 系统设计相关失败的初始概率）
  - *contradicts* thesis §1 (partial, constructive) [med]: 论题主张当前技术栈"workers 之间的协同失真、长链规划退化…仍缺乏学术层面的系统性验证"。[6] 恰好为协同失真维度提供系统性验证——FC2 占 32.3%，并以 1642 轨迹 + κ=0.88 IAA 量化。该结论不反驳论题，但将"未测试的担忧"转为"已度量的患病率"，论题中"缺乏验证"措辞需更新 [6: §4 FC2; thesis §1]

> *Emerging sub-cluster: Heartbeat-Driven Active Scheduling* — 通过周期性心跳信号触发顶层调度器在认知活动（Plan / Reflect / Recall / Dream）间切换，把单 agent 从被动响应升级为自发主动思考；操作维度是"跨步时序频率"，不同于 §1.3（步内规划深度）与 §1.5（调用前选择）。[8] 不适配既有子簇，放置于父桶 §1；建议增设 §1.7，详见 `contradictions.md §Proposed taxonomy extension`。

- **HSC** [8] (Su, 2026, arXiv preprint, single-author) — 提出 Heartbeat Scheduling Controller：在每一 tick `t_k = k·Δt` 读取联合状态 `s_k = [s_k^int, s_k^env]`，由学习策略 `π_θ(s_t, d_t) → z_t` 选择本拍认知活动 `z_t ∈ {Planner, Critic, Recaller, Dreamer, …}` [8: §III, §IV]。双层调度：宏观层 `π_macro` 决定高层认知任务跳转，微观层 `π_micro` 在给定 macro state 下挑选细粒度动作 [8: §III-A]。Dream Mode 在外部任务空闲时由心跳触发，承担三件事：记忆压缩/语义重索引/剪枝、合成经验生成（假设检验+反事实回放+技能复演）、自主目标设定 [8: §IV-C]。状态增强 `s_t' = φ(s_t, h_t)`（其中 `h_t = Encoder(a_{t-k}, …, a_{t-1})`）把近期动作历史嵌入状态以防重复循环 [8: §V-C]。**关键缺陷：理论与实验完全脱节**——§III-IV 通篇是 LLM agent + 可插拔认知模块 + Dream Mode + 自主学习调度的元认知架构，§VI 实验却是 LSTM 在合成 1,800 天 × 24 小时 × 6 类动作序列上的监督 next-action 预测，没有 LLM、没有任何认知模块的实例化、没有 Dream 触发、没有策略梯度/RL/online learning 步骤 [8: §III-IV vs §VI]。所谓"成功"指标（动作覆盖 6/6、预测分布熵 2.31 vs 真值 2.35 bits、稀有动作 recall 78.3%）实际是模型反向工程作者写在数据生成器里的概率规则，且对任何 baseline（ReAct/Reflexion/AutoGPT/朴素小时频率分类器）均无对比 [8: §VI-A.1, §VI-A.3]。扩类实验（加入"Recalling What is Important"为动作 6）仅给出单日定性图，预测时间错位 4 小时（真值 12 时 → 预测 16 时），无测试集聚合 [8: §VI-B.3]。
  - *competes-with* [6] (MAST) [med]：MAST 的 FM-1.3 步骤重复 (15.7%) 与 FM-1.5 终止条件无感知 (12.4%) 都是"何时停 / 何时再思考"的调度问题；HSC §V-C 状态增强正面对应 FM-1.3，HSC 心跳触发反思正面对应 FM-1.5。但 MAST 基于 1642 真实 MAS 轨迹（κ=0.88），HSC 基于合成 LSTM 实验（无 baseline）——两者在"调度问题真实性"上互证，但 HSC 未将 MAST 失败模式作为评测靶点 [8: §V-C; 6: §4]
  - *orthogonal to* [3] (FLARE) [med]：FLARE 改造单次推理深度（前瞻 MCTS），HSC 改造跨次推理频率（心跳调度）；操作层级不同（步内 vs 跨步），可叠加；HSC 未触及 FLARE 形式化的"贪心策略局限" [3: §3.2; 8: §IV]
  - *orthogonal to* [7] (MCP-Zero) [low]：MCP-Zero 主动驱动工具检索（调用前），HSC 主动驱动认知活动调度（跨步时序）；同属"主动驱动"思路但层级互补，本论文不涉及工具
  - *orthogonal to* [4] (OneFlow) [low]：HSC 在单 agent 范畴讨论调度，与 OneFlow 单 LLM 立场一致——是其时序供给侧的一个延伸方向（但未被 [4] 引用）
  - *orthogonal to* [1] (COS-PLAY), [2] (GoA), [5] (Agent-as-a-Graph) [low]：问题域不同（技能进化 / 多 agent 输出协调 / 工具检索）
  - *builds-on* Reflexion (Shinn et al., 2023), Self-Refine (Madaan et al., 2023), ReAct (Yao et al., 2023) [high]：[8] §II-A/B 显式将三者列为"failure-triggered 或 task-completion-triggered"反思代表，把 HSC 定位为 proactive、心跳周期驱动、可在空闲时介入的扩展

  - *relation to thesis* — 直接命中"常驻 Runtime 的学术对照"研究缺口（thesis Design Context goal 2: Heartbeat + Cron + Reflection），但作为**反证案例**而非正面支撑：本论文是迄今唯一直接以"heartbeat scheduling"为标题的 LLM agent 工作，其实验完全不验证主张，揭示**"主动调度学术稀缺是真实的——稀缺到首篇相关论文实质未做相关实验"**。与 thesis 预判一致（"Heartbeat+Cron 学术稀缺，遇到必须收"），但同时降低"通过文献合成获得设计指南"的预期。可吸收的语言资源：HSC 概念命名（macro/micro 双层调度、Dream Mode 三大功能、复合奖励 `R_total = α·R_external + β·R_internal`）、状态增强 `s_t' = φ(s_t, h_t)` 防重复机制（即便实验未验证，作为 Dolphin 状态机引入近 k 步动作哈希以缓解 MAST FM-1.3 的形式化参考）。证据强度全文 [low]——单作者、未同行评议、合成数据、无 baseline、理论与实验脱钩；不应被后续合成误用为"主动调度有效"的引用 [8: §III-IV vs §VI; thesis Design Context goal 2]

> *Emerging sub-cluster: Broadcast-and-Self-Select MAS Communication* — 主智能体不向特定 sub-agent 派发任务，而是把请求广播到共享 blackboard β，helper agent 自判是否响应并写回独占响应板 β_r；helper 间互不可见。操作维度是"通信拓扑"，与 §1.2（GoA 推理时图路由）、§1.5（调用前类型化检索）、§1.6（失败诊断）、§1.7（时序调度）层级均不同。[9] 不适配既有子簇，放置于父桶 §1；建议增设 §1.8，详见 `contradictions.md §Proposed taxonomy extension`。

- **Blackboard MAS** [9] (Salemi et al., 2026) — 把数据科学问答任务的多智能体通信范式从 master–slave 替换为 blackboard：主智能体（ReAct，T=10 步）在 5 动作集（plan/reason/run_code/request_help/answer）中以 `request_help` 写入共享 blackboard β；每个 cluster 的 file agent 与 search agent 自判可否响应，能则写入独占响应板 β_r，主智能体并行回灌 β_r 后选择"使用或忽略" [9: §3.1, §3.2 + footnote 2]。**端到端：** 在 KramaBench / DSBench-filtered / DA-Code-filtered（合计 ~448 题）× 4 backbone（Qwen3-Coder-30B / Gemini 2.5 Flash & Pro / Claude 4 Opus）上 macro 平均较最强基线（Master–Slave）相对提升 13%–57%；Claude 4 Opus + Blackboard 取 31.43% [9: §4.2 Table 1]。**File discovery（Gemini 2.5 Pro）：** Blackboard F1 0.561 vs Master–Slave 0.513 vs RAG 0.247 [9: §4.2 Table 2]。**Scalability claim：** 在数据湖规模（45–1556 文件）与 Blackboard 相对增益间观测正相关（DA-Code 145 文件 +70%、Astronomy 1556 文件 +211%）——但 Astronomy Master–Slave baseline 仅 3.16%–11.52%，分母极小放大相对增益 [9: §4.2 "Scalability Comparison" + Appendix F]。**Cost：** runtime 132–145 秒/题与 RAG/Master–Slave 相差 < 10%，但 USD/题 ≈ 2.3× RAG / 1.8× Master–Slave [9: §4.2]。**关键缺陷：** (a) "Helper agent autonomy" 实质是 prompt 二选一判断，无分布式控制 / 投票 / 仲裁协议——是命名修辞而非架构属性 [9: §3.2 + Appendix C Figure 6]；(b) 多 helper 响应冲突时主智能体仲裁机制完全不透明，case study 显示 8 helper 中 3 个同时响应 [9: §3 + Appendix E]；(c) 主实验默认聚类（Gemini-2.5-Pro 文件名）被自家消融实验证明劣于 E5-Large 内容嵌入 + KMeans，但未用最佳配置重做主结果 [9: §4.2 vs Table 1]；(d) Master–Slave baseline 是 ReAct 主智能体 + 指名调用，比 LangGraph supervisor / CrewAI hierarchical / OneFlow 单 agent 折叠都弱，"Blackboard > Master–Slave" 不等价于 "Blackboard > 工业最佳实践" [9: §4.1]；(e) DSBench / DA-Code 经手工筛除"无 hint 题目"（253q / 91q），无 inter-rater agreement，外部不可比；(f) DSBench 用 Gemini 2.5 Pro 既做 backbone 又做 LLM-as-Judge，无交叉 judge [9: §4.1 + Appendix A]；(g) cluster 上限 27 / k≤16，远低于企业 5k–10k 规模——broadcast 成本随 helper 数线性放大，论文从未触及 [9: §3 Table 3]；(h) offline file-content 读取与分析成本被均摊到所有题目，未做摊销分析 [9: §3.2 vs §4.2]。
  - *competes-with* [6] (MAST) [med]：MAST 把 master–slave 类 MAS 的协调失败归类为 FC2 Inter-Agent Misalignment（占 32.3%）并量化 14 个 FM 的患病率；[9] 在设计层面给出 master–slave 的替代方案（broadcast + 响应隔离 β_r），主张缓解 FM-2.1 信息隐瞒 / FM-2.4 信息共享错误 / FM-2.6 推理-动作错位。但 [9] 未引用 MAST、未把 14 个 FM 作为评测靶点；"Blackboard 缓解了哪些 FM / 引入了哪些新 FM" 是空白。两者在"master–slave 是结构性瓶颈"上互证，方法论 competes（设计修复 vs 失败分类）[9: §3 footnote 2; 6: §4 FC2]
  - *competes-with* [4] (OneFlow) [med]：OneFlow 论证同质 MAS 可被折叠为单 LLM 顺序执行——但前提是任务不超出单 LLM 上下文。[9] 的 DS-GRU baseline（"把所有文件塞 prompt"，对应 OneFlow 的 folded MAS）在 Qwen3 上仅 1.21%、在 Claude 4 Opus 上仅 4.11%——明确**反证** OneFlow 在"上下文必然溢出的数据发现"场景下的折叠主张。两者并不矛盾，而是边界条件互补：OneFlow 适用于上下文充足任务，Blackboard 适用于上下文必然溢出场景。[9] 未引用 OneFlow [4: §3.1; 9: §4.2 Table 1]

  - *orthogonal to* [3] (FLARE) [low]：[9] 主智能体仍是 ReAct（按 FLARE 是贪心策略），但 [9] 任务核心难点是"信息检索 + 分区路由"而非"长链规划"，FLARE 适用度低；两者层级不同（步内推理深度 vs 跨智能体通信结构）
  - *orthogonal to* [2] (GoA), [8] (HSC), [1] (COS-PLAY) [low]：问题域不同（输出协调 / 时序调度 / 技能进化）
